    如何优雅的维护 Python 项目¶

    Python 作为一门简单易用的动态语言，因其功能强大，扩展性强，跨平台而被很多开发者所喜爱。它在机器学习、人工智能、爬虫技术以及数据分析等领域被广泛使用。但是 Python 同样也有一些不足的地方，集中表现在运行速度慢，GIL 导致无法有效利用多线程，动态类型容易导致代码质量失控等方面。

    作为业务开发者，虽然我们不能解决 Python 语言层面的不足，但是可以在工程和代码层面自主把控。通过遵循一些优秀的实践，可以有效的保证 Python 代码质量。那么，当我们用 Python 来实现业务时，我们可以遵循哪些工程实践来优雅的维护 Python 项目呢？

    一、环境¶

    01/ 使用 Python 3.8 以上\版本¶

    目前，最新发布的 Python 大版本为 3.11，Python 2也已经停止维护更新。在 Python 版本的选择上，建议选择 3.8 以上版本，这样就可以享受较新版本带来的性能、安全性以及各种语法糖。如果没有特殊情况，使用的第三方库也支持 3.10 的话，也可以选择 3.10 版本。

    02/ 使用虚拟环境¶

    很多时候可能需要安装了多个 Python 版本来运行不同时期的项目，不同的 Python 项目依赖相同第三库的版本可能也不一样。使用虚拟环境隔绝 Python 版本，既能解决多个版本并存的问题，也能够让 Python 项目的开发运行环境更干净。例如使用 Python venv 包来创建虚拟环境，命令如下：

    python3 -m venv ./venv      # 在项目目录创建虚拟环境
source venv/bin/activate    # 进入虚拟环境
deactivate                  # 退出虚拟环境

    03/ 使用 Docker¶

    Docker 经过这么多年的发展，在各个场景下的应用也都被验证过了，并且有不少的最佳实践。Docker 官方的口号是 “Develop faster. Run anywhere”，使用 Docker 能帮助我们快速开发部署应用。我们可以将项目运行环境的初始化、项目的依赖、项目的运行配置编写成 Dockerfile，再根据 Dockerfile 编译成镜像，将镜像推送到镜像服务，以便在任何环境都能轻松拉取使用，这样便可以保证项目的开发和线上环境保持一致。

    04/ 使用 requirements.txt¶

    生成并维护好项目的 requirements.txt 文件，已经是 Python 程序员的基本共识了。requirements.txt 文件应该只包含本项目所需要的依赖包，并且每个依赖包都指定具体的版本号。有些程序员直接使用 pip freeze 将整个开发环境下的安装包都导入到 requirements.txt 的做法是不值得效仿的，这样做不仅导致在新的环境初始化时安装一些冗余的依赖，也会给项目维护者带来一些困惑。

    二、代码¶

    01/ 规划好项目结构¶

    不管什么语言的项目，好的项目结构不仅有助于开发分工，减少混乱，还能让新人快速的读懂代码。所以不管你维护的项目是新项目还是老项目，遇到项目结构有不合理的地方，能调整一定要及时调整过来，不然越到后面越不好维护。

    02/ import 排序¶

    很多 Python 程序员都会忽略对 import 的排序，觉得没必要。其实对 import 排序，对读懂代码，维护好项目也是很有帮助的。很多项目都按下面的顺序来 import 包：

    # 基础库import osimport re
# 第三方库import requests
# 本项目库import x.x.x

    如果不想手动排，可以使用 isort 这个工具来自动给 import 排序。

    03/ 类型检查¶

    Python 作为动态类型的语言，编码的灵活性很大，开发效率也高。但是项目到了一定规模后，动态类型带来的维护成本就变高了。所以我们维护的 Python 项目，越早加入类型检查，对后期项目的维护越有利。对于类型检查，我们可以使用 mypy，它功能强大而使用简单。

    pip install mypymypy a.py

    04/ 代码规范¶

    关于 Python 代码规范，可以看另一篇《Git Hook 实现 Python 项目代码规范和提交规范落地》的介绍。

    05/ 测试¶

    一个合格的程序员是会做好自测的，对于后端程序员来说，就是要做好单元测试、模块测试和接口测试。在 Python 开发过程中，单元测试可以使用官方的 unittest 框架。模块测试则可以使用第三方测试框架，例如 pytest。接口测试的工具就有很多选择了，一般都兼具自动化和压测功能，比如 JMeter。如果还涉及到前端功能测试，则可以组织开发进行交叉测试，A 同学开发的功能由 B 同学进行测试。开发自测覆盖的范围越广，交付的产品的质量就越高，也就越能保证项目的进度。

    三、工具¶

    01/ 开发IDEA¶

    能够进行 Python 开发的 idea 有不少，有人喜欢用 PyCharm，有人喜欢用VS Code，但是建议团队统一使用 PyCharm 。团队内部可以复用 idea 的设置，团队成员开发时必须安装一些必备插件，比如 editorconfig、mypy、pylint等。通过统一 idea，统一 idea 的设置来减小代码风格的差异。

    02/ 代码管理¶

    使用 git 来管理代码，已经是行业的主流了。使用 git 的好处也不必过多说明了，特别是利用 git 本地钩子，可以让代码格式化、静态类型检查、单元测试都自动化，通过了这些检查才允许提交，就能进一步的提高项目代码的质量。

    03/ pre-commit¶

    使用 git 本地钩子做的事情多了，我们就需要把这些钩子维护起来，方便团队里其他人使用。pre-commit 就是一个管理 git 本地钩子的工具，可以很方便的配置项目所需要的本地钩子。在项目里添加一个配置文件，就能让团队成员很方便的安装好统一的本地钩子。例如下面这个 .pre-commit-config.yaml 配置文件，配置了检查 git 提交信息格式、isort import 排序、flak8代码格式化以及执行mypy测试本地钩子，只有这些钩子都执行通过后，才能进行 git 提交操作。

    # See https://pre-commit.com for more information
# See https://pre-commit.com/hooks.html for more hooks
repos:
    -   repo: git@github.com:One2r/commit-msg-hook.git
        rev: 'ca2480d83aa7d3ce4341a378d3aad56d111f16bd'
        hooks:
            -   id: commit-msg

    -   repo: https://github.com/PyCQA/isort
        rev: 5.10.1
        hooks:
            -   id: isort
                args:
                    - --profile=black

    -   repo: https://github.com/PyCQA/flake8
        rev: 4.0.1
        hooks:
            -   id: flake8
                args:
                - --exclude=.git,__pycache__,docs,utils
                - --ignore=E203,W503
                - --max-complexity=25
                - --max-line-length=128

     -   repo: https://github.com/pre-commit/mirrors-mypy
         rev: v0.961
         hooks:
             -   id: mypy
                 args:
                 - --ignore-missing-imports
                 - --install-types  # See mirrors-mypy README.md
                 - --non-interactive

    通过以上这些优秀的工程实践，就能让我们优雅的维护 Python 项目了，就能让我们说“人生苦短，我用 Python。动态一时爽，一直用一直爽”了。
